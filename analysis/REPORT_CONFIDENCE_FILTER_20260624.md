# 「信頼度：中」シグナルにおける出来高の薄さを考慮した新シグナル条件の検討

**作成日:** 2026-06-24  
**対象データ:** KABUステーション取得 rt_candles（6/17, 6/18, 6/19, 6/22 の4日間）  
**シミュレーター:** resim_engine_v8（本番アプリ完全一致版）

---

## 1. 現状の問題整理

### 1.1 出来高データの欠損状況

KABUステーション中継スクリプト（kabu_board_relay_v5.4）の現状を分析した結果、以下の構造的問題が判明した。

| 日付 | 総足数 | volume=0 | volume>0 | O=H=L=C足 |
|------|--------|----------|----------|-----------|
| 6/17 | 7,893 | 7,869 (99.7%) | 24 (0.3%) | 99.7% |
| 6/18 | 7,849 | 7,823 (99.7%) | 26 (0.3%) | 99.7% |
| 6/19 | 7,316 | 7,296 (99.7%) | 20 (0.3%) | 99.8% |
| 6/22 | 7,435 | 7,415 (99.7%) | 20 (0.3%) | 99.8% |

出来高が記録されている足は11:30（前場終了）と15:30（大引け）の約定集計分のみであり、ザラ場中の1分足出来高は全て0となっている。これはWebSocketティック蓄積が機能しておらず、RESTフォールバック（板情報APIの`currentPrice`から1分足を生成）で全足が作られているためである。

### 1.2 出来高=0がシグナル検出に与える2つの影響

出来高データの欠損は、**シグナル検出の段階**と**信頼度判定の段階**の2箇所に影響する。

#### (A) シグナル検出段階: 出来高を条件に含むシグナルが発生しない

以下の4種類のシグナルは、`is_volume_confirmed()` を発火条件に含んでいるため、**出来高=0の状態では絶対に発生しない**。

| シグナル | 条件 | 出来高=0時の状態 |
|----------|------|-----------------|
| VWAPクロス上抜け | 価格がVWAP上抜け **かつ** 出来高が平均の1.2倍以上 | **発生不可** |
| VWAPクロス下抜け | 価格がVWAP下抜け **かつ** 出来高が平均の1.2倍以上 | **発生不可** |
| VWAP反発（押し目買い） | VWAP接触後に反発 **かつ** 出来高が平均の1.2倍以上 | **発生不可** |
| VWAP反落（戻り売り） | VWAP接触後に反落 **かつ** 出来高が平均の1.2倍以上 | **発生不可** |

さらに、VWAP自体の計算も出来高に依存している（`VWAP = Σ(典型価格×出来高) / Σ出来高`）。volume=0の足はVWAP累積に寄与しないため、VWAPは11:30の1足分の値で固定される（後場は前場終了時のVWAPがそのまま残る）。

> **つまり、VWAP系シグナル4種は完全に死んでいる。** 本来これらは出来高を伴うブレイクアウトの確認として最も信頼性の高いシグナルだが、現状では一切発生していない。

#### (B) 信頼度判定段階: 全シグナルが「信頼度：中」止まり

`evaluateConfirmation()` は以下の3条件でスコアリングしている。

```
score = sum([volumeConfirmed, trendAligned, momentumAligned])
confidence = "strong" if score >= 3 else "medium" if score == 2 else "weak"
```

出来高が常に0のため `volumeConfirmed` は常に `false` となり、最大でもscore=2で「信頼度：中」に留まる。「信頼度：強」のシグナルは原理的に発生しない。

### 1.3 実際に発生しているシグナル（出来高条件なし）

現在エントリーの根拠となっているシグナルは全て**出来高を発火条件に含まないもの**である。

| シグナル | 出来高条件 | 本来あるべき出来高の役割 |
|----------|-----------|------------------------|
| ダウ理論（直近高値/安値更新） | なし | 高値更新時に出来高急増なら本物のブレイク |
| ゴールデンクロス/デッドクロス | なし | クロス時の出来高増加で信頼性向上 |
| 大台超え/割れ | なし | キリ番突破時の出来高急増で加速確認 |
| RSI逆張り | なし | 底値圏での出来高増加で反転確認 |
| 長いヒゲ | なし | ヒゲ形成時の出来高で売買圧力確認 |
| はらみ線 | なし | 出来高減少で「迷い」の確認 |
| ダブルトップ/ボトム | なし | ネックライン突破時の出来高で確認 |
| 三尊/逆三尊 | なし | ネックライン突破時の出来高で確認 |

**これらのシグナルは出来高の裏付けなしに発火しており、「ダマシ」に対して無防備な状態である。**

## 3. 改善案: 出来高代替フィルターの設計

### 3.1 基本的な考え方

ご指摘の通り、**出来高を取得できていなければ正しいシグナルを出せない**。現状の問題は以下の2層構造である。

1. **VWAP系シグナル4種が完全に死んでいる** → 出来高復旧なしには解決不可能
2. **出来高条件なしのシグナル（ダウ理論等）が裏付けなしに発火** → 代替フィルターで緩和可能

推奨案1・2は「出来高が取得できない場合の代替条件」であり、根本解決ではない。根本解決はWebSocket出来高データの復旧（後述）である。

### 3.2 代替フィルター案とバックテスト結果

出来高データが利用できない現状を前提に、「出来高の代わりに何で裏付けを取るか」を検討した。

| 案 | 代替フィルター | 根拠 |
|----|--------------|------|
| 案1 | 9時台のダウ理論上昇シグナルをブロック | 出来高なしでは寄り付きブレイクの真偽判定不能 |
| 案3 | 損切り後30分間、同一銘柄のダウ理論再エントリーをブロック | 出来高なしの連続ダマシ防止 |
| 案5 | 12時台のエントリーをブロック | 昼休み明けは出来高最薄の時間帯 |
| 案6 | 案1+案5+案3の複合 | 出来高が薄い全時間帯を排除 |

## 4. 根本解決: 出来高データの復旧

### 4.1 復旧すべき理由

代替フィルターはあくまで「出来高なし」の応急処置であり、以下の限界がある。

1. **VWAP系シグナル4種は復活しない**: 出来高を条件に含むため、出来高=0では原理的に発生しない
2. **信頼度「強」は達成できない**: `volumeConfirmed=false`固定のため、score=3に到達しない
3. **時間帯ブロックは有効なシグナルも排除する**: 9時台でも出来高急増を伴う本物のブレイクは存在する

### 4.2 復旧後に期待される効果

出来高データが正常に取得できた場合:

| 項目 | 現状（出来高なし） | 復旧後（出来高あり） |
|------|-------------------|---------------------|
| VWAPクロス上抜け | 発生不可 | 出来高急増時のみ発生 |
| VWAPクロス下抜け | 発生不可 | 出来高急増時のみ発生 |
| VWAP反発/反落 | 発生不可 | 出来高急増時のみ発生 |
| VWAP計算精度 | 11:30の1点で固定 | 全足の累積で正確に計算 |
| ダウ理論上昇 | 出来高チェックなし | 出来高急増で「本物」を確認可能 |
| 信頼度「強」 | 原理的に不可能 | score=3で到達可能 |

### 4.3 復旧方法

kabu_board_relay_v5.4のWebSocket接続が切断状態のまま復旧していない可能性が高い。

1. **WebSocket接続状態の確認**: `ws_connected`フラグの監視ログを追加
2. **自動再接続ロジックの強化**: 切断検知→即座に再接続→ティック蓄積再開
3. **出来高復旧後のシグナル条件追加**: ダウ理論上昇に `is_volume_confirmed()` を追加条件として組み込む

## 5. 推奨する実装順序

### ステップ1（即時・代替策）: 案1を実装

出来高復旧までの暫定措置として、9時台のダウ理論上昇シグナルをブロックする。

```typescript
// realtimeSimEngine.ts への追加条件
const jstHour = parseInt(candleTime.split(":")[0]);
const hasVolume = candle.volume > 0;

// 出来高なし時の9時台ダウ理論上昇をブロック
if (!hasVolume && jstHour === 9 && 
    sig.reason.includes("ダウ理論") && sig.reason.includes("直近高値")) {
  return { symbol, tradeDate, candleTime, action: "none" };
}
```

### ステップ2（短期）: WebSocket出来高データの復旧

kabu_board_relay_v5.4のWebSocket接続を修正し、ザラ場中の出来高データを正常に取得する。

### ステップ3（出来高復旧後）: シグナル条件の強化

出来高データが復旧したら、以下を実装する。

1. **ダウ理論上昇に出来高条件を追加**: 高値更新時に出来高が平均の1.2倍以上の場合のみ発火
2. **信頼度「強」の活用**: score=3のシグナルはロット増加や損切り幅拡大を許可
3. **9時台ブロックの解除**: 出来高確認付きなら9時台も有効化
4. **VWAP系シグナルの復活**: 出来高データがあれば自動的に発生する

## 6. まとめ

| 問題 | 影響 | 対策 |
|------|------|------|
| VWAP系シグナル4種が死んでいる | 最も信頼性の高いシグナルが使えない | WebSocket出来高復旧（根本解決） |
| ダウ理論上昇が出来高チェックなしで発火 | 9時台勝率21%のダマシ多発 | 案1: 9時台ブロック（暫定） |
| 全シグナルが信頼度「中」止まり | ロット制御・リスク管理が機能しない | 出来高復旧後にscore=3を活用 |
| 12時台の薄商いエントリー | 勝率0%の損失 | 案6: 12時台ブロック（暫定） |

**結論:** ご指摘の通り、出来高を取得できていなければ正しいシグナルは出せない。現状ではVWAP系4種が完全に機能停止しており、残りのシグナルも出来高の裏付けなしに発火している。推奨案1・2は出来高復旧までの代替策であり、根本解決はWebSocket出来高データの復旧である。
